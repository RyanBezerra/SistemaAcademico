1. DESCRIÇÃO DO SISTEMA
================================================================================

O Sistema Acadêmico é uma plataforma de gestão educacional inteligente desenvolvida
em Python com interface gráfica desktop (PySide6/Qt6). O sistema fornece um ambiente
completo para:

  • Autenticação segura com três níveis de acesso (Aluno, Educador, Administrador)
  • Busca inteligente de aulas com embeddings semânticos (OpenAI)
  • Dashboard personalizado por perfil de usuário
  • Sistema de recomendação de aulas baseado no histórico
  • Gestão de feedback e avaliação de aulas
  • Gestão de instituições e usuários (admin)
  • Cache em memória para otimização de performance
  • Sistema de logging centralizado


2. OBJETIVOS
================================================================================

  1. Fornecer interface intuitiva para descoberta de conteúdo educacional
  2. Facilitar gestão de aulas e recursos pedagógicos
  3. Oferecer recomendações personalizadas via busca semântica
  4. Permitir feedback estruturado sobre qualidade de aulas
  5. Gerenciar múltiplos perfis de usuário com permissões distintas


3. REQUISITOS FUNCIONAIS (RF)
================================================================================

3.1 AUTENTICAÇÃO E AUTORIZAÇÃO
  RF-01: O sistema deve permitir login com email e senha
  RF-02: O sistema deve validar credenciais contra banco de dados (bcrypt)
  RF-03: O sistema deve exibir dashboard diferente conforme papel do usuário
  RF-04: O sistema deve permitir logout a qualquer momento
  RF-05: O sistema deve permitir cadastro de novos usuários

3.2 GESTÃO DE AULAS
  RF-06: O sistema deve listar aulas disponíveis
  RF-07: O sistema deve permitir busca por texto em aulas
  RF-08: O sistema deve buscar aulas por similaridade semântica (embeddings)
  RF-09: O sistema deve salvar histórico de buscas do aluno
  RF-10: O sistema deve exibir detalhes completos de uma aula

3.3 RECOMENDAÇÕES
  RF-11: O sistema deve recomendar aulas com base no histórico do aluno
  RF-12: O sistema deve usar embeddings para calcular similaridade
  RF-13: O sistema deve exibir lista de sugestões de aulas personalizadas

3.4 GESTÃO DE FEEDBACK
  RF-14: O sistema deve permitir avaliar uma aula (1-5 estrelas)
  RF-15: O sistema deve permitir adicionar comentário escrito no feedback
  RF-16: O sistema deve salvar feedback no banco de dados
  RF-17: O sistema deve exibir histórico de feedbacks do educador

3.5 GESTÃO ADMINISTRATIVA
  RF-18: O sistema deve permitir criar/editar/deletar usuários (admin)
  RF-19: O sistema deve permitir criar/editar/deletar instituições (admin)
  RF-20: O sistema deve exibir relatórios de uso e engajamento (admin)

3.6 GESTÃO DE PERFIL
  RF-21: O sistema deve exibir informações do perfil do usuário
  RF-22: O sistema deve permitir atualizar dados do perfil
  RF-23: O sistema deve exibir histórico de atividades do usuário


4. REQUISITOS NÃO-FUNCIONAIS (RNF)
================================================================================

4.1 PERFORMANCE
  RNF-01: Busca simples deve retornar resultados em < 1 segundo
  RNF-02: Busca semântica deve retornar resultados em < 3 segundos
  RNF-03: Cache deve estar configurado com TTL de 3600 segundos por padrão
  RNF-04: Interface deve responder a interações do usuário em < 500ms

4.2 SEGURANÇA
  RNF-05: Senhas devem ser hasheadas com bcrypt (mínimo 12 rounds)
  RNF-06: Conexões SQL devem usar prepared statements
  RNF-07: Tokens/sessões devem expirar após inatividade
  RNF-08: Acesso à API de IA deve ser autenticado
  RNF-09: Logs não devem conter dados sensíveis (senhas, tokens)

4.3 CONFIABILIDADE
  RNF-10: Sistema deve registrar todas as operações em logs
  RNF-11: Cache deve ter mecanismo de fallback para dados não encontrados
  RNF-12: Aplicação deve se recuperar de falhas de conexão com banco
  RNF-13: Sistema deve funcionar mesmo com API de IA indisponível

4.4 USABILIDADE
  RNF-14: Interface deve ser responsiva em resoluções >= 1024x768
  RNF-15: Fonte deve ter tamanho mínimo legível (12pt)
  RNF-16: Todas as ações críticas devem ter confirmação do usuário
  RNF-17: Mensagens de erro devem ser claras e orientar ação

4.5 MANUTENIBILIDADE
  RNF-18: Código deve seguir PEP-8 (padrão Python)
  RNF-19: Arquitetura deve seguir padrão MVC (Model-View-Controller)
  RNF-20: Componentes devem ter baixo acoplamento e alta coesão
  RNF-21: Configurações devem estar centralizadas em config.py

4.6 ESCALABILIDADE
  RNF-22: Sistema deve suportar até 500 usuários simultâneos
  RNF-23: Banco deve usar connection pooling
  RNF-24: Embeddings devem ser cacheados para evitar chamadas repetidas

6.2 PADRÕES DE PROJETO UTILIZADOS

  • SINGLETON: DatabaseManager, CacheManager, LoggerManager
    → Garante única instância em toda a aplicação
    
  • OBSERVER: Sinais Qt (Signal/Slot)
    → Componentes se registram para eventos e são notificados automaticamente
    
  • STRATEGY: Validadores em utils/validators.py
    → Diferentes estratégias de validação intercambiáveis
    
  • MVC (Model-View-Controller):
    → Model: database.py (acesso a dados)
    → View: ui/*.py (interface)
    → Controller: app.py e core logic

9. MODELO DE DADOS (TABELAS PRINCIPAIS)
================================================================================

USUARIOS:
  - id (PK)
  - email (UNIQUE)
  - senha_hash (bcrypt)
  - nome
  - papel (aluno|educador|admin)
  - instituicao_id (FK)
  - data_criacao
  - ativo

AULAS:
  - id (PK)
  - titulo
  - descricao
  - conteudo
  - criador_id (FK → USUARIOS)
  - data_criacao
  - ativo

AULA_EMBEDDINGS:
  - id (PK)
  - aula_id (FK, UNIQUE)
  - embedding (BLOB)
  - data_geracao

HISTORICO_BUSCAS:
  - id (PK)
  - usuario_id (FK)
  - texto_busca
  - resultado_count
  - tempo_ms
  - data

FEEDBACKS:
  - id (PK)
  - usuario_id (FK)
  - aula_id (FK)
  - estrelas (1-5)
  - comentario
  - data

INSTITUICOES:
  - id (PK)
  - nome
  - cnpj
  - endereco
  - data_criacao

13. SEGURANÇA
================================================================================

  ✓ Senhas: Hashing com bcrypt (≥12 rounds)
  ✓ SQL: Prepared statements (prevenção SQL injection)
  ✓ Logs: Não registram dados sensíveis
  ✓ Cache: TTL para expirar dados em memória
  ✓ Autenticação: Email + senha obrigatórios
  ✓ Autorização: Verificação de papel em cada operação


14. PERFORMANCE
================================================================================

  • Cache em memória (LRU): Busca cache < 100ms
  • Connection pooling: Eficiência em conexões BD
  • Embeddings cacheados: Evita chamadas repetidas à API
  • Índices BD: Busca léxica otimizada
  • Queries otimizadas: Evita N+1 queries


15. TROUBLESHOOTING
================================================================================

ERRO: "Dependências não encontradas"
  Solução: pip install -r requirements.txt

ERRO: "Conexão com banco de dados recusada"
  Solução: Verificar DB_HOST, DB_PORT, DB_USER, DB_PASSWORD em .env

ERRO: "API OpenAI inválida"
  Solução: Verificar OPENAI_API_KEY em .env

ERRO: "Cache não está funcionando"
  Solução: Verificar logs com LOG_LEVEL=DEBUG
